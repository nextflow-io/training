## स्वागत और पृष्ठभूमि

नमस्ते, और Hello Nextflow में वापस स्वागत है। यह भाग पांच है जिसे Hello Containers कहा जाता है। और कोर्स के इस भाग में, हम इस बारे में बात करने वाले हैं कि पाइपलाइन के लिए सॉफ़्टवेयर आवश्यकताओं को कैसे encapsulate किया जाए ताकि पाइपलाइन चलाने वाले लोगों को सॉफ़्टवेयर इंस्टॉल करने के बारे में सोचना न पड़े।

अगर तुम मेरे जितने समय से bioinformatics में काम कर रहे हो, तो तुम्हें शायद वो याद होगा जिसे मैं अक्सर बुरे पुराने दिन कहता हूं, जहां जब तुम किसी और की पाइपलाइन चलाना चाहते थे या उनके काम को replicate करना चाहते थे, तो तुम घंटों या दिनों तक उन सभी विभिन्न सॉफ़्टवेयर टूल्स को इंस्टॉल करने की कोशिश में बिताते थे जो उन्होंने उपयोग किए थे, उन्हीं versions पर, उन्हें अपनी मशीन पर compile करने की कोशिश करते थे, और यह एक दुःस्वप्न था। यह वास्तव में कठिन था।

अगर तुम HPC पर चला रहे हो, तो तुमने शायद environment modules का उपयोग किया होगा जहां sysadmins तुम्हारे लिए सॉफ़्टवेयर इंस्टॉल करने की कोशिश करते थे, जो ठीक था, लेकिन फिर भी अपूर्ण था।

लेकिन अब हमारे पास इसे करने के बेहतर तरीके हैं। Nextflow में विभिन्न सॉफ़्टवेयर कंटेनर तकनीकों के लिए built-in सपोर्ट है। Docker सबसे आम है। वही हम आज उपयोग करने वाले हैं। यह Codespaces में अच्छी तरह काम करता है। यह तुम्हारे लोकल कंप्यूटर पर अच्छी तरह काम करता है और यह क्लाउड में अच्छी तरह काम करता है।

लेकिन Singularity या Apptainer भी, जो HPC सिस्टम पर बहुत आम हैं और प्रभावी रूप से बिल्कुल उसी तरह काम करते हैं। या Podman, Shifter, कई अन्य हैं जो सभी बहुत समान हैं।

एक अतिरिक्त जो थोड़ा समान है लेकिन बिल्कुल नहीं, जिसे Nextflow सपोर्ट करता है वह Conda है। और Nextflow तुम्हारे लिए प्रति प्रोसेस आधार पर Conda environments को मैनेज कर सकता है, जो तुम्हारे अपने Conda environments करने से बहुत बेहतर है। और फिर से, पाइपलाइन के साथ ship हो सकता है।

हम इस अध्याय की शुरुआत कंटेनर तकनीकों और Docker और वे कैसे काम करते हैं, इसके बारे में थोड़ी बात करके करने वाले हैं। और हम पहला आधा हिस्सा सिर्फ Docker में manually करने वाले हैं ताकि तुम समझ सको कि hood के नीचे क्या हो रहा है और यह कैसे काम करता है। क्योंकि यह समझना वास्तव में महत्वपूर्ण है कि Nextflow क्या कर रहा है और जब तुम्हारा workflow execute हो रहा है तो यह क्या कर रहा है, यह कैसे समझें।

तो। चलो अपने Codespaces में जाते हैं। अब मैंने सब कुछ फिर से साफ कर दिया है, लेकिन अगर हम Hello Containers में जाते हैं, तो तुम्हें देखना चाहिए कि हमारी सभी scripts और सब कुछ modules अध्याय के अंत की तरह ही हैं। तो हमारे पास यहां अलग-अलग modules हैं, जो मैंने modules डायरेक्टरी में बनाए थे।

वे अभी भी वहां हैं। उन्हें वहां होना चाहिए ताकि यह चल सके। और workflow और output सभी समान हैं सिवाय इसके कि हमने output publishing path को Hello Containers में बदल दिया है, ताकि तुम्हारी फ़ाइलें उस डायरेक्टरी में समाप्त हों।

हम इसे अभी चला सकते हैं यह जांचने के लिए कि यह काम करता है अगर तुम चाहो, या हम terminal के साथ आगे बढ़ सकते हैं।

## 1. कंटेनर को 'manually' उपयोग करना

हम अपने कंटेनरों को मैनेज करने के लिए Docker का उपयोग करने वाले हैं, और मैं यह जांच सकता हूं कि यह मेरे Codespaces पर इंस्टॉल है "docker -v" करके, जो मुझे version दिखाता है जो इंस्टॉल है और सब कुछ, और यह ठीक से काम कर रहा है।

अब कंटेनरों और Docker में दो अवधारणाएं हैं जो वास्तव में महत्वपूर्ण हैं। एक को image कहा जाता है, और एक को container कहा जाता है। Image पूरे फ़ाइल सिस्टम का snapshot है, अगर तुम चाहो, जिसे तुम उपयोग करोगे, और container चलने वाला environment है। तो तुम एक image का उपयोग करके एक container बनाते हो।

एक बार जब तुम उस container में होते हो, तो यह आमतौर पर एक पूरे ऑपरेटिंग सिस्टम की तरह काम करता है। यह बाहरी दुनिया से कट जाता है। यह बाकी सब चीज़ों से अलग हो जाता है, और यह एक अच्छी बात है। इस तरह हम Nextflow के साथ इतनी अच्छी reproducibility प्राप्त करते हैं।

क्योंकि container के अंदर चलने वाले कार्यों के लिए, वे तुम्हारे लोकल सिस्टम पर किसी भी config फ़ाइलों से दूषित नहीं होते हैं। किसी भी अन्य बाहरी प्रभाव से, वे अपने छोटे sandbox में चलते हैं। फिर फ़ाइलें बहुत, बहुत reproducible तरीके से उत्पन्न होती हैं क्योंकि तुम समान underlying libraries, सभी समान dependencies, हर अलग computing environment पर चलने वाले हर व्यक्ति के लिए बिल्कुल समान सॉफ़्टवेयर का उपयोग कर रहे हो। जो स्पष्ट रूप से मुझे लगता है कि शानदार और अद्भुत है कि यह काम करता है। और आज भी, मेरे दिमाग को उड़ा देता है कि यह संभव है।

## 1.1. कंटेनर image को pull करना

तो हम कुछ Docker images और Docker का उपयोग करने की कोशिश करने वाले हैं, जब तुम इसे अपने सिस्टम पर चलाते हो, तो तुम्हारे कंप्यूटर पर, या इस मामले में, code space में एक docker registry होती है, जो सभी विभिन्न images का ट्रैक रखती है जो अतीत में डाउनलोड और उपयोग की गई हैं, और विभिन्न layers जिन पर वे बनी हैं।

हम देख सकते हैं कि Docker के साथ हमारे पास locally कौन सी images हैं "docker image ls" करके। और इस मामले में तुम देख सकते हो कि यहां कुछ Docker images हैं, जो सभी इस Codespaces को सेट अप करने से संबंधित हैं। सभी dev containers और चीजों से संबंधित हैं। तो तुम्हें उनके बारे में बहुत चिंता करने की ज़रूरत नहीं है, लेकिन जैसे-जैसे हम अधिक images जोड़ते हैं और उन्हें डाउनलोड करते हैं, जैसे-जैसे यह कोर्स आगे बढ़ता है, तुम उस सूची को जांच सकते हो और तुम देखोगे कि local registry इन सभी चीजों का ट्रैक रख रही है जिन्हें हमने pull किया है।

लेकिन हम "docker pull" करके एक नया grab करने वाले हैं। और यह Docker को वेब से एक नई image fetch करने के लिए कहता है।

फिर हम उस container के लिए URI डालते हैं। अब यह एक docker image हो सकती है जिसे तुमने locally बनाया है और फिर इंटरनेट पर push किया है। यह एक image हो सकती है जो किसी और ने बनाई है। Docker images बनाने के कई, कई, कई अलग-अलग तरीके हैं, लेकिन arguably सबसे सरल तरीकों में से एक है उसे outsource करना, और किसी और से इसे तुम्हारे लिए करवाना।

और हम इस tutorial में जो उपयोग करने वाले हैं वह Seqera की एक सेवा है जिसे Seqera Containers कहा जाता है।

अब, Seqera Containers पूरी तरह से मुफ़्त है, और यह open source सॉफ़्टवेयर के एक टुकड़े का उपयोग करता है जिसे हम Wave कहते हैं, जिसे Nextflow के complementary तरीके से containers को मैनेज करने के लिए बनाया गया था। और यह कई सामान्य use cases को handle करता है जो हम Nextflow के साथ निपटते हुए पाते हैं।

यह बहुत आम है कि हमें जिस सॉफ़्टवेयर की ज़रूरत है वह Conda में, Bioconda में, या conda-forge channels या अन्य अधिक domain specific channels में packaged है। और Wave और Seqera Containers उससे images बनाने में वास्तव में अच्छा है।

तो मैं इस web UI पर जा सकता हूं और हम "cowpy" नामक package के साथ mess around करने वाले हैं। तो मैं उस package का नाम टाइप करता हूं जो मुझे चाहिए। यह खोजता है, इसने इसे Python package index पर पाया है, तो मैं इसका उपयोग कर सकता हूं। या अगर मैं थोड़ा और इंतजार करता हूं, तो यह bioconda और conda-forge को खोज रहा है। और तुम देख सकते हो, मैं यहां कोई भी conda channel निर्दिष्ट कर सकता हूं। तो अगर तुम Nvidia channel या कुछ और खोजना चाहते हो, तो वह भी काम करना चाहिए।

और फिर मैं निर्दिष्ट कर सकता हूं कि मैं चाहता हूं कि यह मेरे लिए एक docker image बनाए या एक singularity image और यह भी कि मैं कौन सी CPU architecture चाहता हूं। तो amd64 या arm64।

और एक बार bioconda परिणाम सूचीबद्ध हो जाने पर, मैं अब सभी विभिन्न versions भी देख सकता हूं जो उपलब्ध हैं। मैं उसे डालने वाला हूं। और अब मैं खोजना जारी रख सकता हूं और Conda से अधिक packages प्राप्त कर सकता हूं अगर मैं चाहूं और इस container को जैसे चाहूं compose कर सकता हूं, लेकिन मुझे बस वही एक चाहिए। तो मैं Get Container पर क्लिक करने वाला हूं।

अब, किसी और ने पहले से ही समान container का अनुरोध किया है और यह एक registry से वापस आ गया है, तो हमें यह तुरंत मिल जाता है। लेकिन अगर किसी और ने कभी इस सॉफ़्टवेयर package या सॉफ़्टवेयर packages के इस combination के लिए नहीं पूछा होता, तो Wave और Seqera Containers इसे हमारे लिए on the fly बना देते।

हम इस URL को copy कर सकते हैं और हम view build details भी देख सकते हैं। और यह हमें दिखाता है कि सेवा ने backend पर क्या किया। इसने एक conda environment फ़ाइल बनाई। एक docker फ़ाइल, और फिर यह है, docker build प्रक्रिया चला रहा है। इसने एक scan भी चलाया, एक security scan, तो तुम कोई भी CVEs देख सकते हो। और यह तुम्हें बताता है कि यह कब बनाया गया था।

Wave और Seqera Containers इससे बहुत अधिक कर सकते हैं, लेकिन यह एक सरल use case है, जो सबसे आम है। और मुझे कहना चाहिए कि ये images कम से कम पांच साल के लिए hosted हैं। तो तुम इन URLs को अपनी pipelines में बना सकते हो और जान सकते हो कि वे जल्द ही कहीं नहीं जाने वाले हैं।

तो मेरे पास cowpy के लिए मेरी docker image के लिए मेरा URL है।

मैं अब "docker pull" उस URL को कर सकता हूं, और यह सभी विभिन्न layers को fetch करेगा और इस image को डाउनलोड करेगा ताकि यह मेरे लिए locally उपलब्ध हो।

## 1.2. cowpy को one-off command के रूप में चलाने के लिए container का उपयोग करना

ठीक है, अब चलो वास्तव में इसका उपयोग करने की कोशिश करते हैं। तो अब मैं "docker pull" के बजाय "docker run" command का उपयोग करने वाला हूं, और मैं flag "--rm" का उपयोग करने वाला हूं, जो सिर्फ Docker को बताता है कि जब यह मैंने जो पूछा है वह करना समाप्त कर ले तो इस container को shut down कर दे। और फिर मैं container के लिए identifier डालता हूं, जो सिर्फ एक URI है।

और फिर अंत में, मैं वह command निर्दिष्ट करता हूं जो मैं चाहता हूं कि Docker इस image से उत्पन्न container के अंदर चलाए। मैं सिर्फ cowpy कहने वाला हूं, जो उस टूल का नाम है जो Conda Forge से इंस्टॉल है, जो image के अंदर उपलब्ध है।

मैं enter दबाने वाला हूं और वहां तुम जाओ। हमने एक सिस्टम पर cowpy चलाया है। हमारे पास एक छोटी गाय है जो हमें कुछ जानकारी दे रही है।

अब ध्यान दो cowpy मेरे local सिस्टम पर इंस्टॉल नहीं है। तो अगर मैं इसे सिर्फ सभी Docker चीजों के बिना चलाता हूं, तो यह कहता है, command not found। तो इसने एक image pull की है। इसने Docker का उपयोग करके एक container बनाया है, और फिर यह उस container में गया है और हमारे लिए यह command चलाया है और हमें output हमारे terminal पर वापस दिया है। बहुत, बहुत बढ़िया।

## 1.3. cowpy को interactively चलाने के लिए container का उपयोग करना

ठीक है, हम अब एक कदम आगे जाने वाले हैं और इस container को interactively चलाने वाले हैं और थोड़ा poke around करने वाले हैं, ताकि हम देख सकें कि container के अंदर क्या हो रहा है।

तो अगर मैं वापस जाता हूं और मैं अपनी run command लेता हूं और मैं अंत में cowpy से छुटकारा पाने वाला हूं, क्योंकि मैं वास्तव में cowpy नहीं चलाना चाहता। मैं एक Bash terminal चलाना चाहता हूं।

और फिर मैं यहां वापस जाने वाला हूं और मैं "-it" करने वाला हूं, जो Interactive और Terminal या TTY के लिए खड़ा है, और मैं enter दबाने वाला हूं।

और अब तुम देख सकते हो कि prompt, जो मैं टाइप करने से पहले का bit है, बदल गया है। यह Codespaces prompt था जहां इसने डायरेक्टरी कही थी, और अब यह base और roots और tmp कहता है। तो मैं अब container के अंदर हूं, और अगर मैं "ls" करता हूं, तो तुम देखोगे कि इस डायरेक्टरी में मैं जो फ़ाइलें देखता हूं वे मेरे workspace में मेरी फ़ाइलों से अलग हैं।

और वास्तव में, मैं अपने local codespaces workspace या Docker container के अंदर अपनी local drive से कोई भी फ़ाइल नहीं देख सकता। Docker container runtime, पूरी तरह से isolated है और यह बाहर host फ़ाइल सिस्टम से कोई भी फ़ाइल नहीं लिख या पढ़ सकता।

हालांकि, मैं वह सॉफ़्टवेयर देख सकता हूं जो container के अंदर इंस्टॉल है और इसे चला सकता हूं। तो मैं cowpy चला सकता हूं और हम cowpy का उपयोग कैसे करें इसके बारे में थोड़ा और देख सकते हैं। यहां मैं "cowpy 'Hello World'" कर सकता हूं और यह बताता है, इसे वास्तव में मेरे quote को एक छोटे speech bubble के अंदर डालने के लिए कहता है। और तुम विभिन्न प्रकार की गायों को भी चला सकते हो, तो यह गाय होना जरूरी नहीं है। तुम "-c" कर सकते हो। और मैं Sweden में हूं, तो मैं एक moose चुनने वाला हूं। बहुत अच्छा। उसे कुछ antlers दिए।

और कई अलग-अलग हैं जिनके साथ तुम खेल सकते हो, जिन्हें तुम training docs में वर्णित देख सकते हो।

## 1.3.4. Container में data को mount करना

ठीक है। यह अच्छा होगा अगर हम अपने फ़ाइल सिस्टम में फ़ाइलों पर cowpy चला सकें।

बेशक, सिर्फ container होना और कुछ भी access न होना बहुत उपयोगी नहीं है। यह safe और reproducible हो सकता है, लेकिन यह बहुत उपयोगी नहीं है।

तो हम यह कैसे करते हैं? मैं exit टाइप करके इस Docker container से बाहर निकलने वाला हूं, और तुम देख सकते हो कि prompt हमें बताता है कि हम अब फिर से अपने regular Codespaces में वापस हैं।

और मैं फिर से वही command चलाने वाला हूं। लेकिन इस बार मैं यहां वापस कुछ अतिरिक्त flags जोड़ने वाला हूं। और महत्वपूर्ण एक "-v" है, जो एक volume को mounting करने के लिए खड़ा है, जो मूल रूप से एक disk space का हिस्सा है।

"-v" दो भाग लेता है: एक string की तरह है और फिर एक colon और एक string। और पहला भाग local फ़ाइल सिस्टम है, जिसे container में mounted किया जाना चाहिए। और फिर दूसरा भाग है जहां वह container के अंदर समाप्त होना चाहिए।

अब मैं सिर्फ यहां अपना पूरा local फ़ाइल सिस्टम load करना चाहता हूं। तो "." current working directory है। तो मैं सिर्फ "." करने वाला हूं और फिर ":", और फिर हम उसे container के अंदर एक नई डायरेक्टरी में डालने वाले हैं जिसे "my_project" कहा जाता है। इसे वास्तव में कुछ भी कहा जा सकता था।

और फिर मैं फिर से चलाने वाला हूं।

working directory में जहां मुझे dump किया गया है, जो /tmp है, फ़ाइलें वहां नहीं हैं। लेकिन अगर मैं "ls my_project" करता हूं, तो वहां हमारे पास है: वही सभी फ़ाइलें जो हमारे पास locally Codespaces पर थीं अब container के अंदर उस path पर उपलब्ध हैं।

यह read और write access है तो मैं इस डायरेक्टरी में नई फ़ाइलें बना सकता हूं और वे मेरे host फ़ाइल सिस्टम पर दिखाई देंगी। यह विशेष डायरेक्टरी, फिर बिल्कुल वैसे ही व्यवहार करती है जैसे मैं container के बाहर था तो मैं अब read और write और चीजें कर सकता हूं।

## 1.3.5. Mounted data का उपयोग करना

ठीक है, चलो बस साबित करते हैं कि हम यह कर सकते हैं। मैं "cat /my_project/data/greetings.csv" करता हूं। अगर तुम्हें याद है तो इस फ़ाइल की contents इस तरह दिखती हैं। मैं अब इसे cowpy में pipe कर सकता हूं और गाय उस फ़ाइल के विभिन्न outputs को अपने छोटे speech bubble में print करेगी, जो थोड़ा मजेदार है।

## 2. Nextflow में containers का उपयोग करना

तो यह containers का उपयोग करना वास्तव में बढ़िया है। उम्मीद है कि यह समझ में आता है। और तुम इन containers के मूल्य को देख सकते हो और यह analysis सॉफ़्टवेयर चलाने के लिए क्यों उपयोगी है।

लेकिन हम Nextflow के अंदर यह पूरी समान प्रक्रिया कैसे करते हैं? हम खुद loads of Docker commands नहीं चलाना चाहते। हम सिर्फ Nextflow को हमारे लिए यह सब handle करने देना चाहते हैं।

तो चलो इसके माध्यम से काम करते हैं। हम cowpy चलाने के लिए अपनी पाइपलाइन में एक नया प्रोसेस जोड़ने वाले हैं। ठीक है, तो चलो अपने नए प्रोसेस के लिए एक नया module बनाते हैं। तो modules में जाओ, चलो इसे cowPy.nf कहते हैं, और फिर मैं यहां training material से code copy करने वाला हूं।

अब बेशक मैं अब "--character" भी pass कर सकता हूं। विभिन्न options क्या हैं? मुझे लगता है कि एक Turkey है? तो मैं character Turkey का उपयोग कर सकता हूं। यह बिल्कुल उसी तरह चलने वाला है। मैंने dash resume का उपयोग करने का एक और अवसर गंवाया, और अब अगर हम अपनी फ़ाइल load करते हैं और अब हमारे पास एक Turkey है। शानदार।

## 2.3.4. निरीक्षण करें कि Nextflow ने containerized task को कैसे launch किया

ठीक है। अंतिम छोटी चीज। चलो बस जल्दी से इस command को फिर से चलाते हैं, इस बार resume करते हैं, और work directory में एक त्वरित नज़र डालते हैं यह देखने के लिए कि Nextflow हमारे लिए यह सब काम करने के लिए hood के नीचे क्या कर रहा है।

इस बार यह super fast है, चलो इस work directory में जाते हैं, cd work/। अब अगर तुम्हें याद है तो हमारे पास यहां कुछ dot फ़ाइलें हैं और जिसमें हम इस मामले में रुचि रखते हैं वह है जिसे मैंने कहा था कि हमें लगभग कभी देखने की ज़रूरत नहीं है, जिसे .command.run कहा जाता है।

अगर मैं code dot command run करता हूं, तो यह इसे editor में खोलने वाला है। और मैं इस फ़ाइल में search कर सकता हूं और अगर मैं नीचे scroll करता हूं तो मुझे Docker run दिखना चाहिए। और तुम देख सकते हो कि Nextflow हमारे लिए docker run command कर रहा है, जब Docker config में enabled है। इसके यहां कई अलग-अलग flags और चीजें हैं, लेकिन तुम "-v" flag देख सकते हो जिसका हमने खुद उपयोग किया था जब हम चला रहे थे। और तुम देख सकते हो कि यह local workspace directory को container में mount कर रहा है, ताकि container हमारी input फ़ाइलों को access कर सके और outputs को save कर सके। और फिर अंत में, यह .command.sh भी चला रहा है, जो generated script है, जिसमें cowpy command है।

और इसलिए तुम देख सकते हो कि Nextflow workflow logic ले रहा है, जो वह चीज है जिसकी हम वास्तव में परवाह करते हैं, जो हमारे analysis के लिए specific है, और यह सभी clever behind the scenes चीजें कर रहा है ताकि Docker हमारे सिस्टम पर काम करे।

और यह इसे वास्तव में portable तरीके से कर रहा है ताकि पाइपलाइन का एक end user उस technology को switch out कर सके जिसका वे उपयोग कर रहे हैं: Docker, Singularity, Apptainer, Conda। यह वास्तव में पाइपलाइन logic के लिए मायने नहीं रखता, लेकिन Nextflow सभी underlying infrastructure needs को handle करेगा, ताकि यह कहीं भी चले।

और यह वास्तव में Nextflow की superpower है। Reproducibility और portability है। और Nextflow के साथ तुम वास्तव में अपना workflow share कर सकते हो और अन्य लोग इसे अपने सिस्टम पर चला सकते हैं और यह बस काम करेगा।

यह करना वास्तव में, वास्तव में कठिन चीज है, और अब तुम जानते हो कि अपने workflows के साथ यह कैसे करना है।

ठीक है, इस अध्याय के लिए बस इतना ही। अगर तुम कोर्स के अंत में जाते हो, तो तुम्हें कुछ containers के बारे में फिर से एक quiz मिलेगी। उम्मीद है कि यह सब समझ में आया। यह analysis के साथ काम करने का वास्तव में बढ़िया तरीका है। और अगर तुम containers के लिए नए हो, तो मुझे उम्मीद है कि मैंने तुम्हें आश्वस्त किया है कि यह जाने का तरीका है, और तुम कभी पीछे मुड़कर नहीं देखोगे।

लेकिन इसके साथ, शायद थोड़ा break लो, और तुम कुछ मिनटों में मेरे साथ Hello Nextflow के अंतिम भाग छह से गुजरने के लिए शामिल हो, जो सभी configuration के बारे में है।
